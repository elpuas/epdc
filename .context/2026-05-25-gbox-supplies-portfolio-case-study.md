## OG Image Investigation and Fix (2026-05-25)

### Scope
- Investigated why portfolio detail pages were falling back to the global default OG image instead of using `image` from MDX frontmatter.

### Findings (Root Cause)
- SEO metadata is generated by `BaseHead.astro`.
- Portfolio detail pages render through `ProjectLayout.astro`, which is used by language-specific routes:
  - `src/pages/es/crafted-with-love/[slug].astro`
  - `src/pages/en/crafted-with-love/[slug].astro`
  - `src/pages/it/crafted-with-love/[slug].astro`
- `BaseHead.astro` had default fallback behavior:
  - `image = '/default.png'`
- Build verification showed `og:image` and `twitter:image` were outputting the fallback value (`/default.png`) on portfolio pages, even when frontmatter image existed.
- Localized entries (`.es`, `.en`, `.it`) were not the cause; all locales exhibited the same fallback behavior.
- Image paths in frontmatter were valid and already used successfully elsewhere on the page (featured image + JSON-LD), proving source data was present.

### Implementation
- Introduced a dedicated SEO prop in `BaseHead.astro` to avoid ambiguous behavior:
  - Added `ogImage?: string`
  - Added resolution logic: `const resolvedImage = ogImage ?? image ?? '/default.png'`
  - Updated OG/Twitter tags to use `resolvedImage`
- Updated portfolio layout to pass the frontmatter image explicitly via the SEO-specific prop:
  - `src/layouts/ProjectLayout.astro`
  - `<BaseHead title={title} description={description} ogImage={image} />`

### Verification
- Rebuilt site locally with `npm run build`.
- Verified rendered tags in generated files now use entry-specific absolute URLs, for example:
  - `dist/es/crafted-with-love/plataforma-web-multilenguaje-turismo-experiencial-es/index.html`
    - `og:image: https://elpuasdigitalcrafts.com/portfolio/cra-laptop.webp`
    - `twitter:image: https://elpuasdigitalcrafts.com/portfolio/cra-laptop.webp`
  - `dist/es/crafted-with-love/gbox-supplies-plataforma-catalogo-cotizacion-personalizada-es/index.html`
